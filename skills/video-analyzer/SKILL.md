# Video Analyzer

Локальный анализ видео для продакт-менеджера. Принимает **путь к файлу** или **YouTube-ссылку**,
извлекает кадры и речь полностью локально (ffmpeg + WhisperX, без облака), и выдаёт анализ,
адаптированный под жанр видео.

## Жанры (выбирай стратегию под жанр)

| Жанр | Признаки | Фокус анализа |
|---|---|---|
| **Созвон / интервью** | локальный файл, несколько говорящих, диалог | речь: транскрипт + диаризация → продуктовое саммари |
| **Демо инструмента / конкурента** | чаще YouTube, один голос/закадр, показ экрана | **визуал**: кадры → разбор продукта (это не диалог) |
| **Гибрид** | созвон, где клиент шарит экран | связать «что говорит» с «что показывает» |

## Workflow

### 0. Проверка зависимостей (один раз)
```bash
command -v ffmpeg >/dev/null && command -v yt-dlp >/dev/null && [ -d "$SKILL_DIR/.venv" ] \
  && echo OK || echo "RUN: bash $SKILL_DIR/scripts/setup.sh"
```
Если не OK — предложи запустить `bash $SKILL_DIR/scripts/setup.sh` (поставит ffmpeg/yt-dlp и whisperx,
спросит токен HuggingFace для диаризации).
Для PDF-отчёта нужны ещё `pandoc` и `weasyprint` (brew) — проверь `command -v pandoc weasyprint`.

### 1. Рабочая папка (временная, безопасная)
```bash
WORK="$(mktemp -d)" && chmod 700 "$WORK" && echo "$WORK"
```

### 2. Получить видео (URL или локальный файл)
```bash
VIDEO="$(bash "$SKILL_DIR/scripts/ingest.sh" "<URL-или-путь>" "$WORK" | tail -1)"
```

### 2b. Решить: всё видео или отрезок (фокус-режим)
Перед извлечением определи длительность и выбери охват. **Не спрашивай без нужды** — лишний вопрос
на коротком ролике раздражает.
```bash
DUR_MIN=$(( $(ffprobe -v quiet -show_entries format=duration -of csv=p=0 "$VIDEO" | cut -d. -f1) / 60 ))
```
Правило выбора:
- **Пользователь уже назвал отрезок** («первые 5 минут», «с 35-й минуты», «последние 2 минуты»,
  «момент про X на 12:30») → бери его, **не спрашивай**, выставь `FOCUS_START`/`FOCUS_END`.
- **Видео короткое (≤ ~15 мин) и отрезок не назван** → разбирай **целиком**, не спрашивай.
- **Видео длинное (> ~15 мин) и отрезок не назван** → спроси через `AskUserQuestion`: смотреть всё
  или конкретный фрагмент. Поясни, что по фрагменту разбор точнее и быстрее (кадры гуще, транскрипт
  только по окну). Варианты: «Всё видео», «Указать отрезок» (тогда уточни тайминги), при необходимости
  «Только первые N минут». Если пользователь выбрал отрезок — переведи в `FOCUS_START`/`FOCUS_END`.

Сохрани выбор в переменные окружения для шага 3 (пусто = всё видео):
```bash
# пример, если выбран отрезок:
FOCUS_ENV=(FOCUS_START=35:00 FOCUS_END=38:00)   # иначе FOCUS_ENV=()
```

### 3. Извлечь кадры (по смене сцены) и аудио
```bash
bash "$SKILL_DIR/scripts/extract.sh" "$VIDEO" "$WORK"
# → $WORK/audio.wav, $WORK/frames/frame_XXXX.jpg, $WORK/frames/timestamps.txt
```
Кадры берутся по **детекции смены сцены**: на «говорящих головах» их мало, на демо/шаринге экрана —
кадр на каждое переключение UI. `timestamps.txt` (поле `pts_time`) даёт таймкод каждого кадра.

**Фокус-режим (разбор отрезка видео).** Когда на шаге 2b выбран отрезок — задавай окно через env
`FOCUS_START` / `FOCUS_END` (форматы `SS`, `MM:SS`, `HH:MM:SS`; можно только `START` = «до конца»,
только `END` = «с начала»):
```bash
FOCUS_START=35:00 FOCUS_END=38:00 bash "$SKILL_DIR/scripts/extract.sh" "$VIDEO" "$WORK"
```
Кадры **и** аудио режутся по окну: на 3 минутах те же `MAX_FRAMES` дают в разы гуще покрытие, чем
размазанные по часу, а локальный whisper транскрибирует только окно (намного быстрее на длинных
записях). Таймкоды кадров и транскрипта остаются **абсолютными** (реальная шкала видео) — `extract.sh`
прибавляет смещение и пишет его в `$WORK/focus_offset`, а `transcribe.sh` (шаг 5) подхватывает файл
сам и сдвигает SRT/JSON/TXT. Ничего дополнительно передавать в transcribe не нужно. Для длинного
видео фокус по нужному отрезку гораздо полезнее разреженного прохода по всей записи.

### 4. Определить жанр
Эвристика → озвучь предположение пользователю, дай поправить:
- источник YouTube + мало речи/один голос → **демо**;
- локальная запись, много кадров-переключений + речь → **гибрид**;
- локальная запись, мало кадров, диалог → **созвон**.

### 5. Транскрибировать (для созвона/гибрида; для чистого демо — опционально)
```bash
# созвон/гибрид с метками говорящих:
bash "$SKILL_DIR/scripts/transcribe.sh" "$WORK/audio.wav" "$WORK" diarize ru
# демо без диалога (или если речь — просто закадровый комментарий):
bash "$SKILL_DIR/scripts/transcribe.sh" "$WORK/audio.wav" "$WORK" plain ru
```
Транскрипт с таймкодами/метками: `$WORK/audio.srt`, `$WORK/audio.txt`, `$WORK/audio.json`.
Метки `SPEAKER_00/01...` сопоставь с ролями (клиент/интервьюер/PM) по контексту разговора.

### 6. Прочитать ключевые кадры
Используй встроенный инструмент `Read` на файлах `$WORK/frames/frame_XXXX.jpg` (он читает JPG/PNG как
изображения — отдельный vision API не нужен). Для демо смотри все кадры переключений UI; для созвона —
кадры из сегментов шаринга экрана. Сопоставляй кадр с моментом речи по таймкодам.

### 7. Синтез — вывод по жанру (см. ниже)
Сохрани полный транскрипт рядом с источником: `<имя-видео>.transcript.md`
(таймкоды + метки говорящих). Затем выдай саммари.

### 7b. Собрать PDF-отчёт (ПО УМОЛЧАНИЮ для любого жанра — созвон, демо, гибрид)
После разбора **всегда** собирай PDF (не только для созвона). Шаблон зависит от жанра
(см. «Формат: PDF-отчёт»): для созвона — диалог по ролям, для демо — слайды + разбор без диалога.
```bash
# 1) ключевые кадры → постоянная папка (НЕ temp), переименуй осмысленно
mkdir -p "$OUTDIR/assets" && cp "$WORK/frames/frame_00NN.jpg" "$OUTDIR/assets/<имя>.jpg"
# 2) (только созвон/гибрид) черновик диалога из размеченного SRT:
python "$SKILL_DIR/scripts/srt_to_dialogue.py" "$WORK/audio.srt" > "$OUTDIR/_dialogue_draft.md"
# 3) модель пишет $OUTDIR/report.md по шаблону жанра, затем:
bash "$SKILL_DIR/scripts/make_report.sh" "$OUTDIR/report.md" "$OUTDIR/assets" "$OUTDIR/<имя>.pdf"
```

### 8. Очистка
```bash
rm -rf "$WORK"
```
(Не удаляй сохранённый транскрипт, постоянную папку `assets/` и собранный PDF/отчёт.)

## Форматы вывода

### Саммари — созвон / интервью
Контекст пользователя: PM, ведёт Discovery и анализирует работу бизнес-аналитиков и других продактов.
- **Участники и роли** (по диаризации + контексту)
- **Темы** разговора
- **Боли / потребности / задачи клиента** — с цитатами и таймкодами
- **Ключевые решения и договорённости**
- **Action items** (кто / что / когда, если прозвучало)
- **Открытые вопросы и упущенные зацепки**
- **Оценка качества интервью**: открытые vs наводящие вопросы, давали ли клиенту говорить,
  что стоило доспросить (полезно для разбора работы бизнес-аналитика)

### Саммари — демо инструмента / конкурента (визуальный анализ)
- **Что за продукт и для кого**
- **Ключевые экраны / фичи** — со ссылками на кадры и таймкоды
- **UX-флоу и паттерны**; что выделяет продукт
- **Выводы для нас**: чему поучиться, чего избегать

**Покрытие кадров (важно для слайд-видео/демо):** `extract.sh` теперь сам добирает кадры,
если их мало (повтор со сниженным порогом сцен, затем равномерная выборка — см. шаг 3).
**Прочитай КАЖДЫЙ уникальный слайд/экран, а не выборку** — почти-дубли (один и тот же слайд)
пропускай, но не пропускай слайды с новым содержанием. Сверяйся по номерам слайдов (если видны):
пропуск в нумерации = есть непрочитанный слайд, вернись и прочитай его. Не объявляй «больше
смысловых кадров нет», пока не прошёл все уникальные экраны.

**PDF по умолчанию (демо):** после разбора собери PDF (`make_report.sh`) по демо-шаблону —
обложка + разбор простым языком + слайды. Диалога по ролям тут нет (это закадр одного-двух
ведущих), вместо него — связный текст. Транскрипт закадра (`plain`) можно процитировать в выводах.

**Скриншоты — ВНУТРИ текста, рядом с выводом, который они подтверждают** (а НЕ списком в конце).
Каждый скриншот ставь сразу под тем абзацем/пунктом, который он иллюстрирует, с подписью-связкой
(«вот этот экран показывает, что…»). Один вывод — один-два подтверждающих кадра. Так читатель
видит аргумент и доказательство вместе. Финального раздела-свалки «Скриншоты» быть не должно.

### Гибрид
Саммари созвона + врезка «**Что показали на экране**» по сегментам шаринга (кадр + что говорилось).

### Формат: PDF-отчёт (главный осязаемый артефакт, по умолчанию для любого жанра)
Собирается через `scripts/make_report.sh` (pandoc + weasyprint). Цель — чтобы отчёт читал любой
человек без погружения в продукт. Ниже подробно шаблон **созвона**; для **демо** — упрощённый
шаблон в конце раздела (без диалога). Три обязательных части для созвона/гибрида:

1. **Выводы понятным языком, без терминов и усложнений.** Пиши как для коллеги не из темы:
   объясняй механику простыми словами («система ждёт 10 минут, потом проверяет, заказал ли
   человек»), а не ярлыками (никаких «frequency cap / дедуп / атрибуция / gap» — если термин
   нужен, тут же расшифруй обычными словами). Структура свободная, но обычно: «О чём видео» →
   разбор по простым шагам → «Главное» → «Что нового / что уже знали».
2. **Скриншоты** ключевых экранов (кадры из сегментов шаринга) — **встроены рядом с выводом,
   который подтверждают** (не отдельным списком), с короткими понятными подписями-связками.
3. **Расшифровка по ролям** — диалог, где видно, кто говорит.

**Честность по говорящим (важно, всегда соблюдай):**
- Если диаризация **не запускалась/упала** — пиши «реплики восстановлены по смыслу», не выдавай
  за разметку по голосу.
- Если диаризация прошла, но **шумная** (часто так: несколько человек в одной комнате на одном
  микрофоне → их не разделить; удалённый участник через сервис созвона отделяется лучше) — сведи
  метки к реальным людям по смыслу и **по роду глаголов** («выбрала», «зашла» → женщина), а в
  плашке «Как читать» честно опиши: кого отделили уверенно, а кого пишем общим «коллега».
- Никогда не приписывай конкретному человеку реплику, в которой не уверен, без оговорки.

**Шаблон `report.md`** (markdown для pandoc; обложка/плашки — fenced-блоки `::: …`):
```markdown
::: cover
# <Заголовок понятным языком>

Короткий подзаголовок одной строкой

<p class="meta">Команда · дата · кто участвовал (кто в комнате, кто на связи)</p>
:::

## О чём это видео
Простыми словами: что смотрели и зачем.

## <Разбор по шагам / по сценариям>
Механика обычными словами, по шагам.

## Главное
Один-два ключевых вывода без терминов.

## Что нового, а что уже знали
- **Новое:** …

![Подпись-связка: этот экран показывает названный вывод.](имя-файла.jpg)

- **Уже знали — подтвердилось:** …
- **Совпало с нашими решениями:** …

# Расшифровка по ролям

::: notebox
**Как читать.** <честная плашка про состав и про то, как делили по голосам.>
:::

<p class="legend"><b>Участники:</b> …</p>

::: dialogue
| Кто | Реплика |
|-----|---------|
| [Полина]{.p} | … |
| [Валентин]{.v} | … |
| [Коллега]{.k} | … |
:::
```
Имена в первом столбце — через спаны классов: `{.p}` (удалённый/ведёт экран, зелёный),
`{.v}` (основной собеседник, синий), `{.k}` (прочие/коллеги, фиолетовый). Стили — в
`scripts/report.css`. Картинки клади в `assets/` и передавай папку вторым аргументом
`make_report.sh` (он вшивает их в PDF).

**Шаблон `report.md` для ДЕМО** (без диалога; скриншоты ВСТРОЕНЫ рядом с выводами):
```markdown
::: cover
# <Что за продукт/демо понятным языком>

Короткий подзаголовок

<p class="meta">Источник · дата · кто ведёт</p>
:::

## Что это и для кого
Простыми словами.

![Подпись-связка: что на этом экране и почему он тут.](slide-title.jpg)

## Как это работает
### <Шаг/точка роста 1>
Вывод обычным языком.

![Подпись: вот этот экран показывает, что …](slide-01.jpg)

### <Шаг/точка роста 2>
Следующий вывод.

![Подпись: …](slide-02.jpg)

## Главное
1–2 вывода (можно без картинки, если уже показаны выше).

## Выводы для нас
Чему поучиться, что забрать в бэклог.
```
Скриншот ставь **сразу под выводом, который он подтверждает**, с подписью-связкой. Покрой все
уникальные смысловые экраны (не выборку), но раскидай их по тексту, а не свали в конец. Блоки
`::: notebox`/`::: dialogue` для демо не нужны.

## Заметки
- Всё считается **локально**; видео/аудио никуда не отправляются. Токен HF нужен только для
  разовой загрузки модели диаризации.
- Транскрипция идёт через faster-whisper с пословными таймкодами, диаризация — через pyannote
  напрямую (см. `scripts/diarize_transcribe.py`); медленное wav2vec2-выравнивание whisperx НЕ
  используется. Спикеры назначаются по перекрытию слов с речевыми интервалами pyannote.
- **Диаризация прожорлива по памяти и капризна. Запускай строго ОДИН процесс.** pyannote грузит
  весь файл сразу; на 16 ГБ RAM при паре свободных гигов на записи ~20+ мин её **убивает по OOM** —
  в логе при этом НЕТ трейсбэка, процесс просто исчезает (exit 137/144 = SIGKILL, не краш кода).
  Признак: текст распознался (есть «слов с таймкодами: N»), а на фазе «диаризация (pyannote)…»
  процесс умер. Лечение: не запускать второй прогон параллельно (две модели разом = гарантированный
  OOM), освободить память, гнать в один поток и ждать (фаза pyannote идёт после распознавания текста,
  на ней RSS ~1–3 ГБ). Если упала — честно скажи пользователю и предложи повтор или режим `plain`.
- **Качество диаризации зависит от записи.** Несколько человек в ОДНОЙ комнате на одном микрофоне
  pyannote не разделяет (даст «лишних» спикеров и порежет фразы); удалённый участник через сервис
  созвона (Pachka/Zoom) идёт отдельным каналом и отделяется лучше. Поэтому метки `SPEAKER_xx` почти
  всегда надо сводить к реальным людям вручную — по смыслу и роду глаголов (см. «Честность по
  говорящим»). `scripts/srt_to_dialogue.py` даёт черновик (склеивает реплики одного спикера).
- **PDF-отчёт собирается через pandoc + weasyprint** (brew; ставятся `scripts/setup.sh`). Скрипт
  `scripts/make_report.sh` делает markdown → автономный HTML (`--embed-resources` вшивает картинки
  base64) → PDF. Стили — `scripts/report.css`. weasyprint шумит warning'ами про `box-shadow`/`@media`
  — безобидно. Проверка: `mdls -name kMDItemNumberOfPages -raw OUT.pdf`.
- Модели скачиваются автоматически при первом запуске (кэш `~/.cache/huggingface`), поэтому
  первый прогон дольше. Модель Whisper по умолчанию `small` — на этой машине (Apple Silicon,
  ctranslate2 int8) только она работает с приемлемой скоростью (~0.17× реального времени) и при
  этом чисто распознаёт русский. Модели крупнее патологически медленные даже при свободной RAM:
  `medium` ≈ 32× реального времени, `large-v3-turbo` ≈ 4.5×. Менять смысла нет, пока не сменится
  окружение (GPU/оптимизированный билд ctranslate2). Переопределение: `WHISPER_MODEL=...`.
- YouTube требует куки браузера — `ingest.sh` по умолчанию берёт их из Chrome
  (`YTDLP_COOKIES_BROWSER=chrome|safari|firefox|none` для смены/отключения).
- Пороги кадров настраиваются: `SCENE_THRESH` (0.3), `MAX_FRAMES` (80) перед вызовом `extract.sh`.
